# Governance Documents Center — Web-native contract

## Purpose and release gate

Governance Documents is a private internal-document workspace for signed Web
administrators. It provides draft authoring, separated review, retention and
confidentiality metadata, immutable revision snapshots, and a redacted
lifecycle audit.

The module is disabled by default. It is available only when both flags are
enabled:

- WEBAPP_ADMIN_ERP_ENABLED=true
- WEBAPP_GOVERNANCE_DOCUMENTS_ENABLED=true

The flags do not grant authority. Every API read verifies a signed Web admin
session, and every write additionally verifies CSRF, a live session inside the
SQLite transaction, optimistic revision, idempotency, DLP and the applicable
permission rule.

## Explicit Bot boundary

The following Bot source concepts are intentionally TELEGRAM_ONLY:

- admin docs and internal docs commands
- internal_documents data and Telegram attachment/file references
- internal document search and legal export

The Web module never reads or writes those records. It also does not call a
Bot bridge, provider, PayOS, Xu/wallet ledger, job, asset, notification,
external publish, file upload/download or export system. It is not legal
advice and does not auto-publish, auto-notify, auto-delete or auto-enforce a
legal retention outcome.

## Allowed taxonomy

Only these departments and document types may be created:

- marketing: campaign_plan, content_caption, posting_schedule, kpi_report,
  brand_asset
- tech_codex: codex_task, deployment_note, bug_report, architecture_doc
- legal_policy: terms, privacy, data_policy, ip_policy, customer_notice

Metadata is constrained to:

- retention label: manual_review, 3_years, 5_years, permanent
- confidentiality: internal, confidential, restricted

The labels are operational metadata only. They do not create a legal hold,
automatic deletion, legal determination or external publication.

## Lifecycle and review separation

Lifecycle is draft → in_review → approved → archived.

- A document owner may update and submit only its own draft.
- A different signed administrator must approve or reject an in_review
  document.
- Reject requires a bounded review note and returns the document to draft.
- The owner may archive only an approved document.
- Restore moves an archived document to a fresh draft revision.
- There is no hard delete route.

Every successful mutation creates a version snapshot, a redacted lifecycle
event and a generic audit event containing only local document ID, action,
state and request ID metadata. Generic audit never contains title, body, tags,
review notes, secret-like input or reviewer identity.

## Data safety and client behavior

The server rejects secret-like values, credentials, bearer tokens, private
keys, Telegram file references and filesystem-path patterns in all free-text
fields. Browser forms for authoring, lifecycle and filters are marked
no-transient so they are not placed in generic draft persistence. The Portal
fails closed on a missing flag, signed admin session, policy or route-specific
projection; it never substitutes a Bot document list or prior cached body.

Private routes are no-store and are excluded from PWA caching:

- /admin/governance
- /admin/governance/documents/{uuid}
- /api/v1/admin/governance/*

## Web routes

- /admin/governance — summary, create draft, filtered list and pagination
- /admin/governance/documents/{uuid} — record detail, draft editor,
  lifecycle controls, version history and redacted audit timeline

The UI exposes only transitions returned by the server projection. Each
lifecycle action needs an explicit checkbox confirmation and a server-known
acknowledgement phrase generated by the client integration; the browser never
sends an admin ID, Telegram ID, role or canonical permission assertion.

## Focused verification

- tests/test_copyfast_governance.py exercises flags, signed admin and CSRF,
  DLP, malformed IDs, idempotency, revision conflicts and reviewer separation.
- tests/test_governance_portal_contracts.py checks Portal routing,
  non-transient forms, route-fenced hydration, write payload protections,
  no-Bot boundary, no-cache service worker entries and server page guards.
