# Project Package — private Web-native export contract

## Purpose and boundary

`Project Package` is an immutable ZIP snapshot created from one signed
customer's Web Project and its active Studio Documents. It is a real,
server-created output, but it is intentionally **not** a Telegram Bot job,
Bot asset, service package, provider request, payment, Xu ledger entry or
PayOS operation.

| Surface | Owner | Explicitly not used for |
| --- | --- | --- |
| `/projects/{id}/packages` | Standalone Web App | Bot jobs, provider execution, charge, payment or delivery state |
| `/project-packages` | Standalone Web App | Bot `/packages` catalog or membership entitlement |
| Asset Vault references | Standalone Web App | Copying source blobs into a package ZIP |
| `/jobs`, `/assets` | Canonical Bot companion | Project Package status or download |

The server owns snapshot creation, package state, ZIP creation, private file
path and integrity checks. The browser supplies only a Project UUID and an
idempotency key through a signed session plus CSRF.

## Release gates and persistence

The capability defaults to off:

```text
WEBAPP_PROJECT_PACKAGE_ENABLED=false
WEBAPP_PROJECT_PACKAGE_ROOT=
WEBAPP_PROJECT_PACKAGE_MAX_MB=20
WEBAPP_PROJECT_PACKAGE_QUOTA_MB=100
```

When enabled in production, `WEBAPP_PROJECT_PACKAGE_ROOT` must be an absolute
child directory of the Web service's actual Railway volume (`RAILWAY_VOLUME_MOUNT_PATH`
or `/data`). It must never be a static directory, the volume root itself, or
the Asset Vault directory/parent. The application fails closed at startup if
that boundary is invalid. Local/test storage is isolated alongside the
configured Web session database.

The service worker caches only named public shell resources. It never caches
`/api/v1/project-packages`, package downloads, Asset Vault, API, wallet,
payment, admin or other private data.

## API contract

```text
GET  /api/v1/projects/{project_id}/packages
POST /api/v1/projects/{project_id}/packages
GET  /api/v1/project-packages
GET  /api/v1/project-packages/{package_id}
GET  /api/v1/project-packages/{package_id}/download
```

`POST` accepts a bounded JSON object:

```json
{ "idempotency_key": "project-package-..." }
```

It requires signed session, CSRF and an active owner-scoped Project. The
server records an immutable source snapshot and state history
`queued → processing → completed` only after it has atomically promoted a ZIP
into private storage and rechecked its byte size and SHA-256. Failures record
`failed`; a changed/missing/tampered artifact becomes `unavailable` and is not
downloaded.

All responses use the standard envelope. Public package metadata excludes
storage key, hash, filesystem path, source snapshot content, account identity,
Telegram identity, secret, provider data, wallet, payment and signed URLs.

## ZIP contents

The ZIP is server-generated with fixed safe paths:

```text
README.md
manifest.json
documents/001-{kind}.txt
...
```

`manifest.json` contains Project content and document metadata that belongs to
the owner. Asset Vault entries are metadata references only; the ZIP never
copies their blobs, IDs, hashes, storage keys, paths or signed download URLs.
Document names in the archive are generated by the server, not taken from a
customer filename.

Downloads are same-origin attachments and recheck owner, state, storage key,
file type, byte size and SHA-256 on every request. They send `no-store`,
`nosniff`, `no-referrer` and `Content-Security-Policy: sandbox`.

## Safety properties

- Project, documents and Asset Vault references are selected inside the same
  owner-scoped snapshot transaction.
- CSRF protects export writes; idempotency is scoped to account + Project and
  returns the original immutable artifact for the same intent.
- ZIP size, document count, reference count and per-account retained artifact
  quota are bounded.
- Audit events record only counts and byte size, never Project/document text,
  source filenames, storage key, hash or artifact contents.
- Private artifact storage is never mounted under `/static` and stale orphan
  staging/generated files are cleaned only after a retention window.
- The implementation imports no Bot bridge and makes no provider, wallet,
  PayOS or billing call.

## Verification

Focused tests cover signed owner scope, CSRF, idempotency, source snapshot
immutability after a later document edit, ZIP content redaction, private
attachment headers, tamper-to-unavailable handling, production volume/root
validation, Bot-bridge absence, Portal capability separation and PWA
non-caching. The implementation is local/test verified only; it is not a
Railway deployment or live-provider claim.
